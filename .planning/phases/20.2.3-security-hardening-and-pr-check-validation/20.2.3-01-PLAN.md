The SERV-01 test spawns `bun run src/server.ts` and polls port 4219 for up to 25 seconds. It fails consistently in CI across all 3 platforms. The test already has a 30s timeout, but the server may not start reliably in CI due to:
- Port conflicts (other tests using nearby ports)
- `freePort()` killing processes that don't exist
- The server importing many modules at startup

Investigate the spawn approach. The fix should make the test more robust:

1. Add error output capture — after the poll loop, if `ready` is false, read stderr from the spawned process and log it to help diagnose future failures:
```typescript
if (!ready) {
  const stderr = await new Response(serverProc.stderr).text().catch(() => "");
  console.error("[SERV-01] Server failed to start. stderr:", stderr);
}
```

2. Use a unique random port instead of hardcoded 4219 to avoid port conflicts with other parallel tests:
```typescript
const TEST_PORT = 14200 + Math.floor(Math.random() * 1000);
```

3. Ensure the process is killed even if the test fails, by moving the kill to a try/finally or adding a `beforeAll`/`afterAll` cleanup.

4. If the test still cannot be made reliable in CI, add an environment guard that skips the test in CI:
```typescript
const isCI = !!process.env.CI;
it.skipIf(isCI)(
  "SERV-01: starts and responds with HTML",
  async () => { ... },
  { timeout: 30000 }
);
```
Add a comment: `// SERV-01: Skipped in CI — integration test requires full server spawn which is flaky under CI resource constraints. Runs locally.`

Prefer making it work first; only skip if spawn reliability cannot be achieved.
