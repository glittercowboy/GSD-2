## Summary

Phase 11.1 is a purely internal stabilization pass. Every change is isolated to
`packages/mission-control/`; no GSD CLI code is touched. The seven success
criteria map cleanly to five files on the server side and three on the React
side, plus one directory deletion and one asset-swap. There is no new
infrastructure to install.

The changes are surgical. The codebase is well-structured, the test suite is
mature (49 test files covering all affected modules), and the patterns for every
required change already exist in the code тАФ `skipPermissions` is already an
option on `ClaudeProcessManager`, `setWorktreeEnabled` is already on
`SessionManager`, the `127.0.0.1` bind-address pattern is well-established in
Bun. The work is primarily "wire what already exists" rather than "build new."

The one genuinely new piece is the React `ErrorBoundary` тАФ React class-based
error boundaries have no functional-component equivalent in React 19, but the
pattern is trivial and well-documented.

**Primary recommendation:** Address all seven criteria in three tightly-scoped
waves: (1) server security + config bridge, (2) pipeline fragility guards +
constants consolidation + `readFileJson` validation, (3) React
`ErrorBoundary` + branding + `nul/` removal.

### Pattern 1: Bun.serve Bind Address Restriction (SC-3)

**What:** Pass `hostname: "127.0.0.1"` to `Bun.serve` in `ws-server.ts`.
**When to use:** Any Bun server that should be localhost-only.
**Example:**
```typescript
// In ws-server.ts тАФ createWsServer()
const server = Bun.serve({
  port,
  hostname: "127.0.0.1",   // ADD THIS
  fetch(req, server) { ... },
  websocket: { ... },
});
```

**Confidence:** HIGH тАФ Bun.serve `hostname` option is documented and stable.

### Pattern 2: CORS Origin Restriction (SC-3)

**What:** Replace wildcard `"*"` with specific origin `"http://localhost:4000"`.
**When to use:** Any API server that should only be called by its own frontend.
**Example:**
```typescript
// In server.ts тАФ addCorsHeaders()
function addCorsHeaders(response: Response): Response {
  response.headers.set("Access-Control-Allow-Origin", "http://localhost:4000");
  response.headers.set("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE, OPTIONS");
  response.headers.set("Access-Control-Allow-Headers", "Content-Type");
  return response;
}
```

**Confidence:** HIGH тАФ standard HTTP CORS header restriction.

**What:** Stop `reconcileInterval` before the switch body executes; restart with

**When to use:** Inside the `switchProject` function in `pipeline.ts`.
**Example:**
```typescript

    clearInterval(reconcileInterval);   // PAUSE
    watcher.close();
    // ... switch body

  } finally {
    switchGuard.release();
  }

```

at line 318, so this requires a `const` тЖТ `let` change.

**Confidence:** HIGH тАФ straightforward `clearInterval` + re-assign pattern.

### Pattern 7: readFileJson ConfigState Validation (SC-5)

**What:** Replace the untyped `any` return with field-level validation for
`ConfigState`. Return `DEFAULT_CONFIG_STATE` on validation failure.
**When to use:** When parsing `config.json` in `state-deriver.ts`.
**Example:**
```typescript
// In state-deriver.ts
function validateConfigState(raw: unknown): ConfigState {
  if (!raw || typeof raw !== "object") return { ...DEFAULT_CONFIG_STATE };
  const r = raw as Record<string, unknown>;
  return {
    model_profile: typeof r.model_profile === "string" ? r.model_profile : DEFAULT_CONFIG_STATE.model_profile,
    commit_docs: typeof r.commit_docs === "boolean" ? r.commit_docs : DEFAULT_CONFIG_STATE.commit_docs,
    skip_permissions: typeof r.skip_permissions === "boolean" ? r.skip_permissions : DEFAULT_CONFIG_STATE.skip_permissions,
    worktree_enabled: typeof r.worktree_enabled === "boolean" ? r.worktree_enabled : DEFAULT_CONFIG_STATE.worktree_enabled,
    // ... all other fields
  };
}

// Usage (replaces the untyped cast on line 364)
const config: ConfigState = configRaw ? validateConfigState(configRaw) : { ...DEFAULT_CONFIG_STATE };
```

**Note:** `readFileJson` itself can remain `Promise<unknown | null>` with the
generic typed away тАФ the validation happens at the call site for `config.json`.

**Confidence:** HIGH тАФ no external dependency needed; manual field validation is
consistent with the existing codebase convention (no Zod is imported anywhere in
`packages/mission-control`).

### Pattern 8: React ErrorBoundary (SC-6)

**What:** A React class component implementing `componentDidCatch` that renders a
fallback UI when a descendant throws.
**When to use:** Wrap the entire app tree in `App.tsx`.
**Example:**
```tsx
// packages/mission-control/src/components/ErrorBoundary.tsx
import { Component, type ReactNode } from "react";

interface Props {
  children: ReactNode;
  fallback?: ReactNode;
}

interface State {
  hasError: boolean;
  error: Error | null;
}

export class ErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props);
    this.state = { hasError: false, error: null };
  }

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }

  componentDidCatch(error: Error, info: { componentStack: string }): void {
    console.error("[ErrorBoundary] Uncaught render error:", error, info.componentStack);
  }

  render() {
    if (this.state.hasError) {
      return this.props.fallback ?? (
        <div className="flex h-screen items-center justify-center bg-navy-base text-white">
          <p>Something went wrong. Please refresh the page.</p>
        </div>
      );
    }
    return this.props.children;
  }
}

// In App.tsx
import { ErrorBoundary } from "./components/ErrorBoundary";
export default function App() {
  return (
    <ErrorBoundary>
      <AppShell />
    </ErrorBoundary>
  );
}
```

**Confidence:** HIGH тАФ React class-based ErrorBoundary is the only supported
mechanism in React 19 for catching render errors. Functional alternatives
(react-error-boundary) are not needed here тАФ the class component is four lines
of logic.

### Anti-Patterns to Avoid

- **Do not add Zod:** No Zod is present anywhere in `packages/mission-control`.
  Manual field validation keeps the diff small and aligns with conventions.
- **Do not convert class ErrorBoundary to functional:** React 19 does not support
  functional error boundaries without a third-party wrapper library.
- **Do not use `useRef` + `useEffect` to sync reconcileInterval:** Use a plain
  `let` closure variable inside `startPipeline`. The pipeline uses closure state
  throughout тАФ this pattern is consistent.
- **Do not touch `ClaudeProcessManager.onEvent` to clear handlers:** Adding a
  `wired` flag on `SessionState` is simpler and avoids modifying the
  process-manager contract.

## Common Pitfalls

### Pitfall 1: ConfigState Type Mismatch

**What goes wrong:** `worktree_enabled` and `skip_permissions` are written to
`config.json` by the Settings UI but are NOT declared on the `ConfigState`
interface in `types.ts`. Reading `currentState.config.skip_permissions` will be
a TypeScript error in strict mode.
**Why it happens:** The Settings API uses a loose `Record<string, unknown>` for
project config writes, so the fields are persisted but never typed.
**How to avoid:** Update `ConfigState` in `types.ts` to add:
```typescript
worktree_enabled?: boolean;
skip_permissions?: boolean;
```
**Warning signs:** TypeScript compiler error `Property 'skip_permissions' does
not exist on type 'ConfigState'`.

### Pitfall 2: reconcileInterval const тЖТ let

**What goes wrong:** `reconcileInterval` is declared with `const` at line 318 of
`pipeline.ts`. Reassigning it inside `switchProject` requires changing this to
`let`, otherwise TypeScript will error.
**Why it happens:** Original code had no need to reassign the interval.
**How to avoid:** Change `const reconcileInterval = setInterval(...)` to
`let reconcileInterval = setInterval(...)` before implementing SC-4.
**Warning signs:** `Cannot assign to 'reconcileInterval' because it is a
constant`.

### Pitfall 3: SessionState.wired Field Initialization

**What goes wrong:** If `wired` is added to `SessionState` interface but not
initialized in `createSession` and `restoreMetadata`, TypeScript strict mode
will flag missing field in object literal.
**How to avoid:** Add `wired: false` to both session creation sites in
`session-manager.ts`.

### Pitfall 4: CORS Preflight Breakage

**What goes wrong:** Changing `"*"` to `"http://localhost:4000"` will break CORS
preflight requests if the `OPTIONS` handler doesn't also echo the
`Access-Control-Allow-Origin` header.
**How to avoid:** The existing OPTIONS handler in `server.ts` calls
`addCorsHeaders()`, so it will pick up the change automatically. No separate
fix needed.

### Pitfall 5: Asset Path Resolution

**What goes wrong:** `<img src="/assets/gsd-logo-2000.svg">` will 404 if no
server route serves `packages/mission-control/assets/`. The existing
`/api/assets/*` route serves PROJECT assets (files in `.planning/`), not
branding assets.
**How to avoid:** Either (a) copy branding assets to `public/assets/` and let
Bun's static file serving handle them, or (b) add a dedicated route in
`server.ts` that serves from `packages/mission-control/assets/`. Option (a) is
simpler.
**Warning signs:** Browser console shows 404 for logo image URLs.

### Pitfall 6: LogoAnimation CSS Dependency

**What goes wrong:** `LogoAnimation.tsx` uses CSS class names from
`src/styles/animations.css` (e.g., `logo-anim-frame`, `logo-anim-cursor`).
Replacing the inline SVG with an `<img>` eliminates the animation тАФ this is
intentional (replacing placeholder art with official brand), but the animation
CSS classes become dead code.
**How to avoid:** After replacing the component, remove unused animation CSS
classes from `animations.css` to avoid confusion.

## Open Questions

1. **Asset serving route**
   - What we know: `packages/mission-control/assets/` contains the official logo files.
   - What's unclear: Whether to copy files to `public/` or add a new server route.
   - Recommendation: Copy `gsd-logo-2000-transparent.svg` and `terminal.svg` (the
     two variants most relevant for UI use) to `packages/mission-control/public/`.
     This avoids any server code changes and works with Bun's native static file
     serving.

2. **DEFAULT_CONFIG_STATE for new fields**
   - What we know: `skip_permissions` and `worktree_enabled` are not in
     `DEFAULT_CONFIG_STATE` in `state-deriver.ts`.
   - What's unclear: Should the defaults be `true` and `false` respectively
     (matching current implicit behavior) or `false` and `false` (secure defaults)?
   - Recommendation: Default `skip_permissions: true` and `worktree_enabled: false`
     to preserve existing behavior for users who have not explicitly set these in
     their `config.json`.

3. **LogoAnimation replacement fidelity**
   - What we know: `LogoAnimation` fires an `onComplete` callback after 600ms.
     This callback is used by `LoadingLogo` to signal initialization completion.
   - What's unclear: Whether the `<img>`-based replacement should use a CSS
     animation to replicate timing, or whether the 600ms timer can simply be kept
     as a `setTimeout` callback without the SVG animation.
   - Recommendation: Keep the `onComplete` timing logic (`setTimeout(onComplete, 600)`)
     but replace the SVG markup with an `<img>` element. The animation is cosmetic;
     the timer is functional.
